For those who are unfamiliar, Gran Turismo 7 is a racing simulator developed by Polyphony Digital and published by Sony Interactive Entertainment, the latest installment in the Gran Turismo series, released for PlayStation 4 and PlayStation 5 in March 2022.

In 2024, community members discovered that Gran Turismo 7 has a telemetry system broadcast over UDP that allows players to access real-time data about their car's performance. Speed, RPM, gear position, throttle, and more are all available.

### How it works

Gran Turismo 7 transmits data packets over UDP to a specified IP address and port at 60 Hz. User Datagram Protocol (UDP) is a connectionless protocol, fast, lightweight, and well-suited to real-time telemetry. What makes GT7 unusual is that the stream was kept secret since Gran Turismo 6, and it wasn't until 2024 that the community discovered it from old technical documentation. Three things set GT7's UDP stream apart from most games.

### Packet Versions

Since updates 1.42 and 1.68, GT7 sends four distinct packet versions over UDP, referred to as Packet A, B, C, and ~. Each type is an extension of the previous one. The naming comes from the heartbeat value used to request each type.

### Heartbeat

To receive data, you can't just listen, you have to ask. Every 1000 frames, you need to send a heartbeat packet back to the console. This acts as a keep-alive signal and tells the PlayStation where to route the data. The heartbeat value corresponds to the packet type you want; send the wrong value, or nothing at all, and the stream stops or sends garbled data.

The heartbeat is sent on port 33739, while the telemetry data is received on port 33740. These ports are not configurable.

### Encryption

Most games that expose UDP telemetry send plain text. GT7 encrypts its stream using Salsa20, a stream cipher that requires both a key and a nonce to decrypt.

The key is "Simulator Interface Packet GT7 v", the full version string being "Simulator Interface Packet GT7 ver 0.0" truncated to 32 bytes.

The nonce is more interesting. At offset 0x40 in the packet buffer, 4 bytes are read as a 32-bit integer (iv1). A second value (iv2) is derived by XORing iv1 with a constant that depends on the packet version:

```c++
0xDEADBEAF  // Packet A
0xDEADBEEF  // Packet B and C
0x55FABB4F  // Packet ~
```

The 8-byte nonce is then assembled as [ iv2 | iv1 ]. The XOR'd value comes first, then the raw value second. Both halves come from the same 4 bytes, so the nonce is self-referential. The version-dependent constant means the same source bytes produce a different nonce depending on which packet type you're decrypting.

### The Discovery

While developing and maintaining telemetry libraries for Gran Turismo 7, I occasionally noticed strange behavior when experimenting with the heartbeat used to keep the telemetry stream alive. The community was already familiar with Packet A, B, and ~, but some heartbeat values produced responses with lengths that didn’t match any known packet structure.

Most values simply resulted in corrupted data or a fallback packet, but one particular value consistently returned a larger packet of 368 bytes.

To investigate further, I began recording raw packet dumps and comparing them against the known Packet B layout. Since Packet C is an extension of the previous packets, all of the existing fields aligned perfectly, with additional data appended to the end.

The challenge then became identifying what these unknown bytes represented.

### Identifying the New Fields

Finding the meaning of the new data required correlating changes in the packet with actions performed in-game.

I did this through raw data analysis and looking for trends in the values. I printed the data in various formats, including char, uint8, and float.

For example, by driving on different surfaces and comparing captures, I found that certain values changed depending on whether the car was on asphalt (T), grass (G), curbs (C), or snow (s). Timing data appeared to increase continuously throughout a lap and reset when crossing the start line, revealing the current lap time field.

Other fields were identified by the community, such as the wheelbase length, which was cross-referenced against real life values.

### New Data Fields

- Surface type in contact with the car's tires. (char[4])
- Current laptime. (uint32)
- Steering angle of the front two wheels. (float)
- Length of the car wheelbase (measured from the left front wheel to the left rear wheel). (float)
- The car category of the current vehicle. (char[4])

These additions significantly increase the amount of information available to external devices and software. Dashboards can now display live lap timing, motion rigs can react more accurately to steering input, and analysis tools can better understand vehicle behavior and track conditions.

As far as I’m aware, this packet structure had not been documented previously, making Packet C the first major addition to Gran Turismo 7 telemetry discovered by the community since 1.42.
